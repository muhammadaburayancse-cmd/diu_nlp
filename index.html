Section 02

How Tokens Are Generated

The right granularity depends on the task. Choose a tokenization strategy and watch your text transform in real time.

Understanding token IDs, vocabulary, and attention masks:

Vocabulary

A fixed mapping of tokens → integer IDs. Every model has its own vocabulary built during training. Out-of-vocabulary tokens become [UNK].

Attention Mask

A binary array: 1 = real token, 0 = padding. When batching sequences of different lengths, shorter ones are padded and masked so the model ignores pad tokens.

Special Tokens

[CLS] marks start of sequence; [SEP] separates segments. BERT, GPT, and T5 each use different conventions.

Subword Methods

BPE (GPT) merges frequent character pairs. WordPiece (BERT) maximizes likelihood. SentencePiece works on raw bytes — no whitespace needed.

Section 04

Stemming vs Lemmatization

Two strategies, one goal: reduce words to their root. They make very different tradeoffs.

✂ Stemming
  • Chops suffixes using rules — no dictionary lookup
  • Fast, aggressive, language-rule-based
  • May produce non-words (organization → organ)
  • Good for search indexing and rough classification
  • Porter, Snowball, Lancaster are common algorithms
📖 Lemmatization
  • Reduces to dictionary (canonical) form using morphology
  • Slower but accurate — produces real words
  • Requires part-of-speech tag to work correctly
  • ran → run, better → good
  • Essential for semantic search, QA, user-visible text

Section 05

NLTK vs spaCy vs Transformers

Three ecosystems, three philosophies. Pick the one that matches your constraints.

📚 NLTK
  • Best for teaching and research
  • Highly modular — swap any component
  • Extensive corpora and resources
  • Slower at scale, verbose API
  • Penn Treebank POS tags
⚡ spaCy
  • Production-grade performance
  • Unified pipeline: tok → POS → lemma → NER
  • 60+ language models available
  • Hard to swap individual components
  • Universal Dependencies POS tags
🤗 Transformers
  • Use when training/running a transformer
  • Tokenizer tied to each model
  • Subword tokenization (BPE / WordPiece)
  • Classical preprocessing mostly skipped
  • Use AutoTokenizer

import nltk from nltk.tokenize import word_tokenize from nltk.stem import PorterStemmer, WordNetLemmatizer from nltk import pos_tag text = "The cats were running." tokens = word_tokenize(text) # handles contractions, Unicode stems = [PorterStemmer().stem(t) for t in tokens] tagged = pos_tag(tokens) # [('cats','NNS'), ('running','VBG'), ...] def to_wn(tag): if tag.startswith('V'): return 'v' if tag.startswith('J'): return 'a' return 'n' lem = WordNetLemmatizer() lemmas = [lem.lemmatize(t, to_wn(tag)) for t, tag in tagged]

POS Tagging & Morphology

Lemmatization requires grammatical context. Without knowing that "running" is a verb, you can't reliably reduce it to "run". Hover over each word below to see its tag and features.

WordPOSPenn TagMorphological FeatureLemma
Why POS matters for lemmatization

The word better has two lemmas:
• As ADJ → good (comparative adjective)
• As ADV → well (adverb)
Without POS, a lemmatizer must guess. If it guesses wrong, downstream models learn from incorrect features.

Section 07

Preparing Text for ML Models

After linguistic preprocessing, text must become numbers. There are several strategies, each with a different information density.

Bag of Words

Count of each vocabulary word per document. High-dimensional sparse vector. Loses word order entirely.

TF-IDF

Term frequency × inverse document frequency. Downweights common words. Still sparse; still no word order.

Word2Vec / GloVe

Dense 100–300 dim vectors trained on cooccurrence. Captures semantic similarity. One vector per word type.

Contextual Embeddings

BERT / GPT produce different vectors for the same word in different contexts. "Bank" ≠ "Bank" anymore.

Section 09

The Two Failure Modes Nobody Warns You About

Most tutorials stop at the algorithms. These two issues will silently destroy production accuracy.

📦

Reproducibility Drift

NLTK and spaCy change tokenization behavior between versions. What produced ["do","n't"] in spaCy 2.x may produce ["don't"] in 3.x. Your model was trained on one distribution. Inference runs on another. Accuracy quietly degrades and nobody knows why.

Fix: Pin library versions in requirements.txt. Write a preprocessing regression test that freezes expected tokenization of 20 sample sentences. Run it on every upgrade.

🔀

Train/Inference Mismatch

Train with aggressive preprocessing (lowercase, stopword removal, stemming), deploy on raw user input, watch performance crater. This is the single most common production NLP failure.

Fix: If you preprocess during training, you must run the identical function during inference. Ship preprocessing as a function inside the model package — not as a notebook cell the serving team later rewrites.
